# 测试防腐：让测试别成为重构的枷锁

好测试在重构时为你亮红灯并指出问题；坏测试在改文案时全挂、在真出事时绿得发慌。两者的差别在于**测行为还是测实现**。本文讲测试怎么腐烂、怎么防。

## 一、6 种腐烂类型

| 腐烂类型 | 症状 | 防腐写法 |
|---|---|---|
| 硬编码清单 | 断言里写死一整个数组/对象，新增一项就挂 | 断结构与关键字段，不枚举全量 |
| 断言私有实现 | 断内部 state、调用次数、私有方法 | 断用户可感知的公共行为 |
| 过时期望值 | 期望值无业务语义注释，改了分不清是演进还是回归 | 期望值绑业务语义注释 |
| 桩与重构脱节 | mock 了内部模块，重构后 mock 还在骗自己 | 只 mock 不稳定边界（HTTP），不 mock 内部 |
| 字段对齐漂移 | fixture 结构与后端真实响应不符，测试绿但线上错 | fixture 对齐真实响应形态，最好与契约同源 |
| 边界过宽 | 组件加载就拉起重子树，逐层 mock 追不完 | 缩小被测边界，下沉单元或上抛 E2E（见 `source-pushback.md`） |

核心：**测试咬死公共行为而非私有实现。** 实现是重构时必变的，行为相对稳定；绑死实现 = 测试变枷锁。

## 二、结构性断言优于枚举值断言

```ts
// ❌ 硬编码清单：列表多一项就挂，且看不出意图
expect(result).toEqual([
  { id: 1, name: 'A' }, { id: 2, name: 'B' }, { id: 3, name: 'C' },
]);

// ✅ 结构性断言：表达"我关心什么"
expect(result).toHaveLength(3);
expect(result[0]).toMatchObject({ name: 'A' });
expect(result.every(x => x.id > 0)).toBe(true);
```

## 三、防腐三道闸

1. **增量门禁**：只跑牵连测试快速暴露腐烂——`vitest related <改动文件>`，比全量快得多，本地改完即跑。
2. **skip 必带 TODO + 复活条件**：`it.skip` / `it.todo` 不能裸放，必须注明为什么 skip、什么条件下复活，否则永久腐烂在那。
3. **可测性是默认产出**：不是写完功能再"补测试"，而是开发时就保证可测——交互元素有语义锚点、逻辑有可观察出口。可测性差时反推改源码，见 `source-pushback.md`。

## 四、测试债务量化（可选）

测试成为阻力往往是债务信号。监控两个指标：

- **测试脆弱度**：过去 30 天内**无代码变更却失败**的测试数——越高说明测试越耦合实现细节。
- **测试维护比**：每新增 100 行业务代码，需要改动的测试代码行数。> 1:1 说明测试耦合过深，该重构。
