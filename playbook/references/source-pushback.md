# 反推源码：测不动往往是源码在报警

这是 playbook 最高价值、也最容易缺位的一环。普通流程把测试当**验收终点**——spec 跑绿就完事。但很多时候**单价值最高的改动是测试逼出来的源码修复**。

## 核心心法：测试是架构探针

> 测试写得顺，说明源码健康；写不出或写得脆，往往是源码在报警。

playbook 不为了凑覆盖率硬写垃圾测试。当某个对象**测不动**——加载就崩 / 必须 mock 一大坨才跑得起来 / 只能断言私有字段 / E2E 只能退到 xpath 和动态 className——**停止硬写**，按下表判断病灶，产出《测试阻塞报告》反推改源码，而不是交一个脆弱测试。

## 信号表：测试现象 → 源码病灶 → 反推动作

| 测试现象 | 源码病灶 | 反推动作 |
|---|---|---|
| 加载顺序触发崩溃/undefined，运行期（webpack/vite）却正常 | 循环依赖 TDZ 环——打包器解析顺序碰巧绕过，单测不同加载顺序把它暴露了 | 断环：把绕回的 import 内联进唯一消费方；基础常量文件不 import 重链路 |
| 组件加载期就崩、逐层 mock 追不完 | 边界过宽：基础组件传递依赖拉起重子树（图表/富文本/重 SDK） | 拆分组件 / 纯逻辑下沉单元 / 整体上抛 E2E |
| 必须断言私有字段才能验证 | 职责未拆，逻辑塞在 service 内部无可观察出口 | 抽独立 Controller / 纯函数，让逻辑可单测 |
| E2E 只能退到 xpath / nth-child / 动态 className | 交互节点缺语义锚点（裸 `div onClick`、无可访问名） | 补语义元素 + 可访问名 / `<模块>-<语义>` testid（治 a11y 同时治可测性） |
| fixture 要写一大坨嵌套才跑得通 | 接口响应结构过度耦合 | 推后端契约收敛，或前端加适配层隔离 |

注意：选择器优先级规约（见 `selectors.md`）只覆盖最后一类（E2E 缺锚点）。**前几类是单测/集成层才会暴露的**——这正是"只测 E2E"会漏掉的盲区。

## 《测试阻塞报告》格式

产出后**停下等用户拍板，不擅自改源码**：

```
## 测试阻塞：<被测对象>
- 尝试层级：<单元 / 集成 / E2E>
- 卡点现象：<具体报错 / 为什么会脆>
- 判断病灶：<对照上表的源码问题>
- 反推建议：<怎么改源码让它可测，影响面与风险等级>
- 临时方案：<能否降层覆盖部分逻辑 / 是否先 skip + TODO 标注复活条件>
```

## 反推纪律

1. **先分清是测试腐烂还是真实 bug**：实现变了测试没跟，是腐烂（修测试，见 `anti-rot.md`）；测试咬住源码缺陷，是反推。
2. **反推改动独立成任务/commit**，按风险矩阵走，不混进测试 commit。
3. **绝不擅自改未改动区域的源码**——反推建议交用户决定。
