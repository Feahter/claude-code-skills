# 测试金字塔：全层分层判断

playbook 不只是 E2E 引擎。接到"给这个模块加测试"时，第一步不是写 spec，而是**判断每个被测对象该落在哪一层**。测错层，又贵又脆。

## 金字塔 vs 测试奖杯

传统后端推金字塔（底层海量单元、顶层少量 UI）。前端项目充斥组件、状态、异步请求，纯函数占比没那么高，**集成层才是性价比核心**——这就是"测试奖杯"模型：

```
        E2E            少量，只测核心赚钱路径
     Integration       ★ 前端重心：组件+状态+网络揉在一起测
       Unit            纯逻辑、工具函数、hooks
      Static           TS + ESLint，编码期就干掉一半低级错误，ROI 最高
```

不必纠结金字塔还是奖杯，记住结论：**前端别把 80% 精力砸在纯单测上，组件一组合到处是 bug；集成层投入产出最高。**

## 分层决策树（口诀）

> **逻辑级写单元，数据流写集成，流程级才上 E2E。**

| 被测对象形态 | 落层 | 边界红线（违反就别在这层硬写） |
|---|---|---|
| 纯函数、金额/格式化/校验/reducer/判定逻辑 | 单元 | 无 |
| 组件 → hook → 接口 → 渲染 的真实数据流 | 集成 | 组件在 jsdom 加载就拉起重子树（图表、富文本、重 SDK）→ 不属本层，下沉单元或上抛 E2E |
| 真实路由 / 登录态 / 完整业务流 / 跨页跳转 | E2E | 涉及真实下单/支付/资金 → mock 掉，或停在"提交按钮就绪可点"绝不点击 |

**一次迭代常跨多层**，不是三选一。改一个交易面板：金额计算 → 单元，面板数据流 → 集成，下单流程 → E2E，三层组合覆盖。

判断标准：**这个 bug 用户能感知到吗？** 只有用户感知得到的才值得进 E2E；用户感知不到的逻辑错（算错金额、格式化挂了）下沉到单元，又快又准。

## 边界拿不准时怎么探

集成对象先 `npx vitest run` 跑一个空壳渲染：
- **加载期就崩** = 边界过宽，组件拉起了重子树 → 回判（下沉单元 / 上抛 E2E），别逐层 mock 硬追。
- 加载顺序触发崩溃但运行期正常 → 多半是源码循环依赖，进 `source-pushback.md`。

## 分层覆盖率门限

覆盖率是**找盲区的工具，不是 KPI**。100% 是毒药——逼你给 getter、日志、纯透传组件写无意义测试。按模块设差异化门限：

| 模块类型 | 行覆盖目标 | 理由 |
|---|---|---|
| Utils / 纯函数 | 95%+ | 输入输出确定，无 excuse |
| Hooks | 90%+ | 逻辑密集 |
| Services / API 层 | 90%+ | 数据正确性命脉 |
| Store / 状态 | 90%+ | 跨组件共享，错了波及面大 |
| Components | 70%+ | 布局代码不测，交互逻辑必测 |
| Pages | 60%+ | 大量胶水，靠 E2E 兜核心流程 |
| 配置 / 类型文件 | 0%（排除） | 无逻辑 |

**看分支覆盖而非只看行覆盖**：一行 `a ? b : c` 行覆盖了，两个分支不一定都走到——bug 常藏在没走的 `else` 里。

> 高风险模块高覆盖、低风险模块低覆盖，而不是全项目一刀切。哪怕整体只有 40%，只要登录/支付/金额这几条 100%，项目就稳。

## 异常优先

线上事故来源是接口超时、500、断网、空数据、权限不足，而不是"登录成功"。被测对象的用例配比向异常倾斜：

```
正常路径   ~30%
异常路径   ~70%
```

核心路径（认证/支付/金额/并发）的错误分支**必测**，且这类地方显式报错优于静默容错。
