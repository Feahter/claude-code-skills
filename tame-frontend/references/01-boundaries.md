# 业务边界与复杂度治理

## 核心命题

**架构的本质不是技术选型，而是控制复杂度。** 一个系统最贵的成本永远不是第一次开发，而是 1 年后的改动、3 年后的重构、10 个团队同时开发、老代码没人敢碰。所以拆分模块的唯一标准是：

> 改 A 时会不会炸 B/C/D？

文件数量、代码优雅、设计模式数量都不是判据。**降低耦合**才是。

## 决策框架

### 按业务领域分包，不是按技术类型分包

技术类型分包是大多数项目的起点：

```
src/
├── components/
├── utils/
├── hooks/
├── services/
└── store/
```

看起来整洁，但业务逻辑到处共享、到处 import、到处调用。半年后任何改动都会全站爆炸。这叫**伪模块化**——目录干净，依赖一团乱麻。

按业务领域分包才是真模块化：

```
src/
├── core/                  # 基础设施：网络请求、底层工具、跨域共享
└── domains/               # 业务领域，自闭环
    ├── order/
    │   ├── api/
    │   ├── model/         # 类型与领域模型
    │   ├── hooks/         # 业务逻辑
    │   ├── components/    # 仅本域复用的 UI
    │   └── index.ts       # 唯一对外出口
    ├── user/
    └── payment/
```

每个 domain：UI / state / api / types / hooks 全部自闭环，对外暴露最小接口。**高内聚 + 低耦合**。

### 三层架构：表现层 / 领域层 / 基础设施层

```
UI (表现层)
  ↕ 仅依赖 hooks/model
domain (领域层)
  ↕ 通过 Adapter 防腐
infra (基础设施)：API、缓存、日志
```

UI 永远不直接碰 API。后端怎么变，前端 UI 基本不动。这是**抗变化能力**——架构价值不在跑起来，而在变化时还能稳。

防腐层的具体做法见 [03-contracts.md](03-contracts.md)。

## Deletion Test：识别假抽象

判断一个模块是不是值得存在，做这个测试：

> 把这个模块删掉，复杂度是消失了，还是分散到 N 个调用方？

- **消失了** → 这是 pass-through，假抽象，删
- **分散到 N 个调用方** → 它在挣钱，留

很多团队为了"可测试性"把纯函数抽出来，结果真正的 bug 藏在调用方，纯函数本身永远不出错。这种抽取没有 locality（局部性）——逻辑分散到调用方了。

参考 `improve-codebase-architecture` skill 的 deepening 思想：好的模块是**深的**——小接口背后藏大量行为；坏的模块是**浅的**——接口几乎和实现一样复杂。

## 模块联邦 / 微前端：何时该上，何时是毒药

模块联邦（Module Federation）是大型团队解耦神器，但**过早拆分是架构毒药**。

### 何时该上

| 信号 | 解释 |
|---|---|
| 多个独立业务域 | 比如电商有商品、订单、支付、营销 4 个独立团队 |
| 团队 ≥ 3 个独立小队 | 部署节奏、技术栈选型、迭代周期不同 |
| 子应用需独立部署 | 上线一个不能拖另一个；事故隔离 |
| 已经在 Monorepo 里跑通逻辑解耦 | 微前端是部署解耦，逻辑解耦应该先做 |

### 何时是毒药

- 单一团队、单一业务域，纯粹"为了用而用"
- 团队 < 5 人，运维成本远大于收益
- 子应用之间共享大量状态 → 微前端反而成累赘
- 没有解决样式隔离、路由冲突、共享依赖版本对齐之前就上

### 推荐路径（小步演进）

1. **第一步：单仓单 app**，按 domain 分包，做好领域边界
2. **第二步：Monorepo**，用 pnpm workspaces + Turborepo 把共享代码抽到 packages/
3. **第三步：业务真正独立后**，再切微前端（Webpack Module Federation / Rspack / Native Federation）

> 警告：跳过第二步直接上微前端，是 90% 团队后悔的决策。

## 分层渲染（壳层 / 内容层 / 交互层）

大型应用的另一种解耦思路——按渲染时机和稳定度切：

| 层级 | 内容 | 渲染策略 |
|---|---|---|
| 壳层（Shell） | 导航、全局状态、权限 | SSR 直出，秒开 |
| 内容层（Content） | 业务模块 | 流式加载、按需激活 |
| 交互层（Interaction） | 复杂表单、实时协作 | 完全 CSR，独立水位 |

每个模块崩溃不会拖垮整个页面。这种 fail-soft 是大型系统的基础。

## 检查清单

把这些问题代入正在评估的代码或方案：

- [ ] 按业务领域还是按技术类型分包？如果是后者，业务变更时改动是不是要跨 4-5 个目录？
- [ ] UI 组件里有没有直接 import API 客户端？有 → 防腐层缺失
- [ ] 一个 domain 内的状态、hooks、API 调用是不是都能在 domain 目录内找到？还是散落在 src/store、src/utils、src/services 里？
- [ ] 有没有"通用 utils"目录在不停膨胀？里面是不是塞了大量业务专属逻辑？
- [ ] 跑一遍 deletion test：哪些"小工具"删掉后，复杂度其实没增加？这些是假抽象
- [ ] 如果团队从 5 人扩到 20 人，现在的分包方式还能让他们并行不冲突吗？
- [ ] 在讨论微前端 / 模块联邦之前，是否先解决了模块边界、状态泄漏、防腐层？

## 反例与代价

| 反模式 | 代价 |
|---|---|
| 按技术类型分包（components/utils/hooks/services） | 业务变更跨多目录改文件，难以并行开发 |
| "通用 utils" 目录膨胀 | 实际是业务专属逻辑伪装成通用，复用率低，删不掉，改不动 |
| 组件直接调 fetch / API | 后端改字段全前端要改 |
| 为可测试性抽取的纯函数，但 bug 都在调用方 | 没有 locality，重构难度反而增加 |
| 单一业务域上微前端 | 运维成本翻倍，没解决任何实际问题 |
| 跨子应用共享大状态对象 | 微前端本意是解耦，结果重新耦合，比原来还乱 |
| 小步重构没做，直接 big bang 重写 | 90% 重写项目失败 |

## 演进路径模板

如果用户来咨询"我现在的项目结构很乱，怎么改"，给阶梯式路径：

1. **冻结新增技术目录**，新增功能必须按 domain 写
2. **画出现状依赖图**，找出耦合最严重的 1~2 个 domain（往往是 user / order / payment）
3. **先抽防腐层**：API 层加 Adapter，把后端字段稳定住
4. **逐 domain 收编**：一次只动一个，把它的 components/hooks/api 收到 `domains/<name>/` 下
5. **新老共存**：老的 components/utils 不删，新代码不许往里塞
6. **半年内自然过渡**：90% 业务收敛后，再决定是否上 Monorepo / 微前端
