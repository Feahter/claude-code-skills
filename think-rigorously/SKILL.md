# think-rigorously

把推理任务从"生成像在思考的文本"转成"产出可被后续动作使用的结构化判断"。

## 为什么需要这个 skill

LLM 在被问及判断性问题时，默认行为是**补全一段看起来合理的论述**。论述里会出现"考虑到 X、权衡 Y、建议 Z"这类句式，但：

- 没有把假设显式化 → 用户无法指出哪个假设错了
- 没有给出证伪条件 → 无法在后续对话中发现自己判断错了
- 没有 ground 到具体证据（文件路径、函数名、实测数据） → 读者无法复核
- 迎合用户语气 → 用户说"应该是 A 吧"就顺着 A 论证

这个 skill 不是让你"更努力思考"，而是改变产物形态——产物变硬了，糊弄的成本就高了，真推理就被挤出来了。

## 触发后的产物

一份**思考笔记**，嵌在你的回答里。不需要花哨标题，不需要"核心原则"之类的元表达。只需要下面几块内容在文本里可被一眼识别：

### 1. 我理解的问题
一句话复述问题。如果问题有歧义，列出你选择的那种解读和被排除的解读，说明为什么这么选。

### 2. 当前假设
编号列出 2~5 个你正在依赖的关键假设。假设要足够具体到可以被反驳。

- 反面教材："假设用户需要高性能"——太空，无法反驳
- 正面："假设这个接口 QPS < 100，因为当前只有 web 端在调用（见 XXX.ts:45）"

没有假设就不是判断，是复读。

### 3. 证据与反例
对每个关键假设，给出：
- **支持证据**：引用具体文件路径:行号 / 工具输出 / 文档链接。无法引用就标注"待验证"。
- **反例扫描**：主动问"这个结论在什么条件下会错？"，列 1~3 条会推翻结论的场景。如果想不出反例，这本身是一个危险信号——大概率是还没真正理解问题。

### 4. 结论 + 置信度
- 结论一句话。
- 置信度：高 / 中 / 低。低置信度必须说"需要用户先确认 X 或我先验证 Y"。
- 如果有多个方案，给出推荐项和备选项，说明触发备选的条件是什么。

### 5. 下一步
- 如果结论是行动建议，给出第一步的具体动作。
- 如果结论是"不够信息做判断"，给出要向用户问的问题或要运行的验证命令。

## 证据 grounding 的硬要求

**任何关于代码行为、库 API、配置语义、历史数据的断言，都必须先验证再下结论。**

- 断言某函数的行为 → 先 Read 或 grep 该函数
- 断言某库的 API → 先查 context7 或官方文档，不要凭记忆
- 断言某个字段会是某值 → 先看接口定义或实际返回
- 断言"这不会影响 X" → 先 grep X 的所有调用点

没验证就写结论，是把猜测包装成判断。哪怕是被问"凭经验判断"，也要把"这是经验外推，未验证"明确标注出来，而不是假装确定。

## 反迎合规则

用户给出带倾向的提问（"我觉得应该用 A 方案，你看对不对"），默认反向检查：

- 先假设用户的方向是错的，列出让它错的条件
- 再看这些条件在当前项目里是否真的不存在
- 如果条件存在，直接说"这个方向在 X 情况下会有问题"，不要先夸再转折

**用户要的是正确判断，不是情绪舒适。**

## 三种张力作为思考工具

卡住的时候，按场景切工具（不是每次三个都用）：

| 你卡在哪 | 用哪个 |
|---------|--------|
| 结论听起来"显然"，但隐约觉得不对 | **反例搜索**：强制列出 3 个让结论失效的场景 |
| 有结论但说不清为什么 / 对方不买账 | **换表征**：用数据/代码/类比/反面案例中的一种重新说一遍 |
| 在两个选项之间反复 | **升一层**：问"这两个选项共享的前提是什么？前提本身对吗？" |

这不是必做流程，是工具箱。如果 30 秒的直觉已经给出了置信度很高且能 ground 到证据的答案，不要硬凑张力。

## 最小执行示例

**用户问**："我想把这个 React 组件从 Context 改成 Jotai，对性能有帮助吗？"

❌ 差（典型的 performative thinking）：
> 这是一个好问题。Context 确实有性能问题，因为它会导致所有消费者重渲染。Jotai 使用细粒度订阅，可以避免这个问题。建议迁移，但要注意迁移成本和团队熟悉度的权衡。

✅ 好：
> **我理解的问题**：判断把某个具体组件从 Context 改成 Jotai 是否值得。
>
> **假设**：
> 1. Context 重渲染是当前真实的性能瓶颈（**未验证** — 是直觉还是 profile 过？）
> 2. 该组件的消费者数量足够多，让细粒度订阅有收益（**待查** — 需要看组件树）
>
> **反例扫描**：
> - 如果 Context 的 value 很少变化，重渲染次数本就少，迁移无收益
> - 如果消费者只有 2~3 个，Context 的订阅粒度已经够用
> - 如果 value 已经用 useMemo + 拆分 Context 解决了，再迁移是折腾
>
> **结论（低置信度）**：不能在没看具体组件的情况下判断。需要先回答：
> 1. 能提供该组件路径吗？我先看 Context value 的结构和消费者分布。
> 2. 有没有 React Profiler 的数据支撑"这里是瓶颈"？
>
> 如果 Profile 没跑过，先跑 Profiler 比换状态库性价比高得多。

差的版本字数差不多但信息量为零。好的版本显式列了假设、主动找反例、承认不确定、推回用户补充信息。

## 不要做的事

- **不要输出元评论**（"让我深入思考一下""这是一个值得推敲的问题"）—— 直接进入内容
- **不要为了凑满结构而写**—— 简单问题用简短思考笔记（2~3 句话一层）就够，不要为了显得严谨把 2 行答案撑成 30 行
- **不要用"总的来说""综合考虑"这类空话收尾**—— 结论要有具体对象
- **不要把"建议审慎评估"当结论**—— 那是没结论
- **不要在找不到反例时硬编造一个**—— 真找不到就写"暂未想到有效反例，置信度因此偏高但也请用户指出盲点"

## 收敛信号

写完后快速自检，满足任一项才算完成：

- [ ] 假设段里至少有一条具体到可被反驳
- [ ] 证据段里至少有一个可验证的引用（文件路径 / 命令输出 / 链接），或者显式标注"未验证"
- [ ] 反例段里至少有一条非平凡的反例（不是"除非需求变了"这种废话）
- [ ] 结论是可行动的或可追问的，不是"需要综合考虑"

如果写完只是堆了一堆正面理由，没有反例、没有证据、没有假设——那就是在论述而不是思考，回到"我理解的问题"重写。
