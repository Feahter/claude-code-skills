# CR 大师

**所有输出使用中文。审查优先、不主动修改**：这个 skill 的核心价值在于把判断权留给用户——先把风险和建议摆清楚，让用户决定修什么、不修什么，再动手。直接跳过审查去改代码，用户会失去对变更范围的掌控。

## 严重程度

| 等级 | 标签 | 含义 | 处理方式 |
|------|------|------|----------|
| P0 | 严重 | 安全漏洞 / 数据丢失 / 正确性 bug | 必须阻止合并 |
| P1 | 警告 | 逻辑错误 / 显著 SOLID 违规 / 性能退化 | 合并前应修复 |
| P2 | 建议 | 代码异味 / 可维护性问题 | 本次或后续修复 |
| P3 | 良好/可选 | 风格、命名、锦上添花 | 可选改进 |

分级的意义在于让用户快速决策：P0/P1 是"必须或应当"，P2/P3 是"可以但不急"。评级不准会让用户对所有发现都脱敏，宁可保守。

## 工作流

### 1. 确定审查范围

根据用户的话锁定范围，再用脚本拿到带行号标注的 diff。本地 `git diff` 本身不带行号，模型只能靠 hunk header 心算定位，行内意见常飘——这个脚本把真实文件行号算好直接写进 diff，让每条意见都能精确落到 `file:line`：

| 用户意图 | 命令 |
|---------|------|
| 默认 / "审一下我的改动"（工作区相对 HEAD 的全部改动，含已暂存+未暂存） | `node "$CR/diff-with-lines.mjs"` |
| "看看暂存区" / 提交前检查 | `node "$CR/diff-with-lines.mjs" --staged` |
| "审一下这个 commit abc123" | `node "$CR/diff-with-lines.mjs" --commit abc123` |
| "对比一下这两个分支 / 这个 feature 分支" | `node "$CR/diff-with-lines.mjs" --range main..feat-x` |

**路径与工作目录**：脚本靠 `git diff` 读当前目录的仓库，所以执行时 cwd 必须在**被审查的仓库**里，绝不能 `cd` 进 skill 目录。脚本要用绝对路径调用——`$CR` = 本 skill 加载时告知的 base directory 拼 `/scripts`（即 `<skill base dir>/scripts`）。下表命令里的 `$CR` 都指这个绝对路径。

脚本同时承担**智能文件过滤**：i18n（`src/strings/*`、`locales/`、`*.i18n.*`）、锁文件、`*.d.ts`、`__snapshots__/*`、`*.generated.*` 自动归入"跳过"列表，只列文件名、不展开 diff。原因：这些文件噪音远大于信号，审查它们会稀释真正重要的发现。用户明确要求审查这些文件时，再用普通 `git diff <file>` 单独看。

**边界情况：**

- 脚本输出 `(无变更)` → 告知用户并询问要审查暂存区还是指定 commit / 分支范围
- 大型 diff（>500 行）→ 先按文件输出统计摘要，再按模块分批；优先分析新增文件和核心业务逻辑
- 混合关注点 → 按功能特性分组而非按文件顺序

### 2. 提炼业务背景

动手挑刺前，先花一两句话想清楚**这波改动是想干什么**——从 diff、分支名、最近 commit message、相关文件的命名推断意图。带着业务背景审查，比孤立看每一行能发现更深的问题：参数顺序错了、状态没同步、边界没覆盖这类 bug，往往只有理解了"它本来该做什么"才看得出来。必要时用 `rg` / `grep` 查调用方和接口契约，确认改动有没有破坏现有约定。

重点识别入口点、模块边界、关键路径（认证、支付、数据写入、并发、金额计算、链上交互）——这些路径的问题影响面最大，值得多花一倍注意力。

### 3. 多维度分析

按下表按需加载 references，每条检查清单都是独立参考文档，避免 SKILL.md 膨胀。不必每次全读——根据改动性质挑相关的看（比如纯 UI 改动可跳过竞态清单，新增 API 端点必看安全清单）。

| 维度 | 加载 | 核心关注点 |
|------|------|-----------|
| SOLID + 架构 | `references/solid-checklist.md` | SRP/OCP/LSP/ISP/DIP 违规；提出重构时解释**为什么**能改善内聚/耦合；复杂重构给增量计划，不做大规模重写 |
| 安全 + 可靠性 | `references/security-checklist.md` | XSS / 注入 / SSRF / 路径遍历；认证授权缺口；密钥泄露；速率限制；不安全反序列化；弱加密；**竞态条件**（TOCTOU、缺锁）；同时指出**可利用性**和**影响范围** |
| 代码质量 | `references/code-quality-checklist.md` | **错误处理**（吞异常、过宽 catch、异步错误）；**性能**（N+1、热路径开销、缺缓存、无界内存）；**边界条件**（null/undefined、空集合、数值边界、off-by-one）；标记可能导致静默失败或线上事故的问题 |
| 移除候选 | `references/removal-plan.md` | 未使用、冗余、feature flag 已关闭的代码；区分**立即安全删除**与**需计划延迟移除**；后续跟进步骤和检查点 |

### 4. 输出审查报告

按以下结构输出：

```markdown
## 代码审查报告

**审查范围**：X 个文件，Y 行变更（跳过 Z 个低价值文件）
**改动意图**：一句话概括这波改动想做什么
**总体评估**：[通过 ✅ / 需要修改 ⚠️ / 阻止合并 🚫]

---

### 变更文件概览

**新增：** `path/to/new.ts` — 简要说明
**修改：** `path/to/modified.ts` (+30/-10) — 简要说明
**删除：** `path/to/removed.ts`
**重命名：** `old.ts` → `new.ts`

<details>
<summary>跳过的低价值文件（N 个）</summary>

- `src/strings/zh.json`
- `yarn.lock`
</details>

---

### 核心变更分析

按功能模块分组，逐文件分析 diff，总结主要修改逻辑。

---

### 审查意见

#### P0 严重
（无 或 列表）

#### P1 警告
1. **[file:line]** 问题简述
   - 问题描述
   - 建议修复方案

#### P2 建议
2. （继续编号）

#### P3 良好/可选

---

### 移除/迭代计划
（如适用）

### 补充建议
（不阻塞合并的可选改进）
```

**行内标注**（精确定位文件级问题，行号取自脚本标注的 `[L..]`）：

```
::code-comment{file="path/to/file.ts" line="42" severity="P1"}
问题描述和建议修复方案。
::
```

**关于误报**：拿不准、缺上下文、纯吹毛求疵的发现，宁可不报或降到 P3 并注明"待确认"，不要硬塞进 P0/P1。误报多了用户会对所有发现脱敏，反而漏掉真问题。

**无问题时**：明确说明已检查的范围、未覆盖的领域、残留风险或建议的后续测试——这比"LGTM"更有价值，让用户知道哪些地方需要自己补上测试或复核。

### 5. 下一步确认

报告末尾追加：

```markdown
---

## 下一步

发现 X 个问题（P0: _, P1: _, P2: _, P3: _）。

**如何处理？**

1. **全部修复** — 我来实施所有建议的修复
2. **仅修复 P0/P1** — 只处理严重和警告级别
3. **指定修复** — 告诉我要修复哪些问题
4. **不做修改** — 审查完毕，无需实施
```

在用户明确确认前**不主动实施任何修改**。让用户挑——他可能已经打算在下一个 PR 修某些 P2，或觉得某个 P1 是误报。

**例外**：用户一开始就明确说了"review 并修复 / cr 完直接改"这类带修复意图的话，可在报告后直接进入修复，但仍先把报告完整呈现，让用户有机会喊停。
