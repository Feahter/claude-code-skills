# CC track · agent prompt 优化

写在代码或对话里的 `Task(prompt=...)` / `Agent(prompt=...)` 字段——发给子 agent 的任务描述。

**能力边界**：这是轻量通用建议，不如 skill-description 深入。主要做一份 checklist 扫，不跑真实 agent 验证。

## 诊断清单（五条）

### ① 背景交代

子 agent 看不到你的会话历史。prompt 里有没有告诉它"我们在做什么、为什么做"？

- **失败表现**：直接一句"修一下这个 bug"，agent 根本不知道 bug 是什么
- **修法**：补一段"Context:" 说明当前目标、已知约束、已尝试过什么

### ② 任务目标 / 输出形态

prompt 明确了 agent 要产出什么形态的结果吗？

- **通过条件**：说清楚"输出一份报告 / 代码 diff / PR 链接 / 一行结论"
- **失败表现**：任务开放式，agent 返回一大段东西你还要再筛
- **修法**：加 `报告在 200 字以内` / `只输出修改后的代码，不要解释` 这类约束

### ③ 输入范围与上下文完整度

agent 要读哪些文件 / 能读到哪些上下文？

- **失败表现**：让 agent "看下整个项目"，它会漫无边际地 grep
- **修法**：给出具体文件路径、行号范围、或"只看 src/x/ 下的 .ts 文件"

### ④ 终止条件

什么时候算"做完了"？

- **失败表现**：没写终止条件，agent 可能无限补充细节
- **修法**：`做到 X 就停，不要继续迭代` / `发现 Y 立即返回不要修复`

### ⑤ 权限与边界

agent 能写文件吗？能 git push 吗？能跑测试吗？

- **失败表现**：让 explore 类 agent 去做写操作，或让执行型 agent 只读
- **修法**：明确 `只读，不写任何文件` 或 `可以修改 X 但不要动 Y`

## 输出模板

```markdown
## 诊断结论

- [① 背景] ❌ 缺失：prompt 里没写当前任务目标
- [② 输出形态] ⚠️ 模糊：说"帮我看看"，没定形态
- [③ 输入范围] ✅ 过：给了具体文件路径
- [④ 终止条件] ❌ 缺失
- [⑤ 权限边界] ✅ 过：明确只读

## 改写版

<原始 agent prompt>

改写后：
Context: 我们在排查 X 模块 Y bug，已经排除了 A/B 两种原因。[① 新增]
任务：找出 Z 函数为什么返回 null，**只输出一句话结论 + 最多 3 条证据**。[② 新增]
输入范围：只看 src/x/y.ts，别 grep 其他文件。
终止条件：找到根因立即返回，不要尝试修复。[④ 新增]
权限：只读。
```

## 常见反模式

| 反模式 | 修法 |
|---|---|
| 一句话任务："修 bug" | 补 Context + 输出形态 + 终止条件 |
| "帮我分析一下代码" | 改成"输出 5 条具体问题 + 每条定位到文件:行号" |
| 假设 agent 知道项目约定 | 把必要约定抄进 prompt，别假设它读过 CLAUDE.md |
| 让 agent "做到满意为止" | 永远给**可验证的**终止条件 |
