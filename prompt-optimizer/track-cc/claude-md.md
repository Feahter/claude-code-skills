# CC track · CLAUDE.md 指令优化

用户/项目/子目录 CLAUDE.md 的指令内容。

**能力边界**：轻量通用建议。核心毛病是"写了但不生效"、"互相冲突"、"被截断"。

## 诊断清单（五条）

### ① 层级冲突

CLAUDE.md 有多层（`~/.claude/CLAUDE.md` 用户级、项目根 `CLAUDE.md`、子目录 `CLAUDE.md`）。不同层是否出现**互斥规则**？

- **失败表现**：用户级写"always respond in English"，项目级写"用中文回答"
- **诊断动作**：询问用户能否贴出相关层的内容，对照扫冲突
- **修法**：互斥规则下沉到最细粒度（项目 > 用户），上层只放**通用偏好**

### ② 总 token 量（被截断风险）

单个 CLAUDE.md 过长会被系统截断（典型阈值 200 行），后半段指令失效。

- **失败表现**：user-level CLAUDE.md 超 200 行，底部写的规则从来没被执行过
- **诊断动作**：读文件 `wc -l`，超 200 行警示
- **修法**：拆分成多个子文档，在 CLAUDE.md 里用 `@path/to/rules.md` 引用

### ③ 具体性

指令是"空洞原则"还是"可执行规则"？

- **失败表现**：`写出高质量代码` / `遵循最佳实践` / `保持简洁`
- **修法**：改成可验证的具体规则：`函数超 50 行必须拆分` / `新代码禁止用 Chakra UI`

### ④ 优先级标记滥用

`IMPORTANT` / `MUST` / `NEVER` 扎堆会让模型脱敏——全都重要 = 全都不重要。

- **通过条件**：强调标记 ≤ 3 处，且用在真正的"硬红线"上
- **失败表现**：每段都有 IMPORTANT，模型反而开始挑着执行
- **修法**：保留 2-3 条硬红线用 NEVER / MUST，其他改用陈述句

### ⑤ 反模式/禁止项缺失

只说"应该做什么"不说"不要做什么"，模型会在灰色地带自主发挥。

- **失败表现**：规范只有 do，没有 don't
- **修法**：每类规则配对"禁止 X"，比如"用 `p-16px` + **禁止**用 `p-16`"

## 常见反模式

| 反模式 | 修法 |
|---|---|
| 整份都是宣言式口号 | 抽具体行为规则，删标语 |
| `IMPORTANT: ALWAYS ...` 满屏 | 留 2-3 处硬红线，其余改陈述 |
| 只写"正确做法"不写"错误示例" | 每条规则配 do/don't 对照 |
| 一份 CLAUDE.md 超 300 行 | 拆文件，用 @path 引用 |
| 项目级重写用户级通用偏好 | 项目级只写项目独有的东西 |
