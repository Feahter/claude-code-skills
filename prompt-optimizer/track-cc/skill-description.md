# CC track · skill description 优化

这是 CC track 里**最深入、最高 leverage** 的资产类型。skill 没触发 90% 的根因都在这一行 description 里。

## 诊断维度（六条）

### A. 触发词覆盖度

description 有没有列出用户真实会说的话？

- **通过条件**：覆盖≥2 种说法（正式说法 + 口语说法，如"代码审查" + "cr 一下"）
- **失败表现**：只写抽象能力名词（"代码审查工具"）不写用户 utterance
- **诊断手段**：列出"用户最可能怎么说"→ 逐条对照 description

### B. 反向排除

有没有写明"不适用于 XX"防止误触发？

- **通过条件**：出现 "不用于" / "不适用" / "skip" / "NEVER" / "跳过" 等排除语
- **失败表现**：只有"做什么"没有"不做什么"——Claude 默认宽松触发，沾边就加载
- **典型模板**：`不适用于：<反例 1>、<反例 2>`

### C. 跨 skill 冲突（按需跑，成本高）

同目录其他 skill 的触发词和本 skill 有没有重叠？

- **诊断动作**：
  1. 读取当前 skill 的上级目录（通常 `~/.claude/skills/` 或 `.claude/skills/`）
  2. 遍历所有兄弟 skill 的 `SKILL.md` 前 20 行，提取 description
  3. 抽取关键词交集，列出"可能争抢的 skill"
- **输出形态**：
  ```
  可能冲突：
  - code-review-expert（重叠词："审查"、"cr"）→ 建议你在本 skill description 里
    加反向排除："不适用于整个 git 变更的结构化审查（那种场景用 code-review-expert）"
  ```

### D. 动作-对象结构

description 是"模糊功能描述"还是"结构化触发条件"？

- **通过条件**：出现 `当用户 <动作> <对象>` 或 `用户说 "X" / "Y"` 这种明确锚点
- **失败表现**：只写"优化你的代码"、"帮助分析"这类形容词堆积
- **改写模板**：
  - ❌ `优化代码质量的工具`
  - ✅ `当用户让你重构超过 50 行的函数、或者明确说"简化这段代码"时触发`

### F. 场景示例

description 里有没有内嵌典型 utterance 做锚定？

- **通过条件**：出现 `Triggers:` / `例：` / `example:` / `用户说 "X"` 等显式样例
- **失败表现**：只有描述没有例子
- **参考**：项目里已经在用这种模式的 skill（如 `rxjs-master`、`code-review-expert` 的 description 结构）

### J. 负样本回放（按需跑，需要用户样本）

用户提供"本应触发但没触发的原话"或"不该触发但触发了的原话"时，做回放验证。

- **触发条件**：用户给出至少 1 条负样本
- **诊断动作**：
  1. 起一个子 agent（subagent_type: general-purpose 即可）
  2. 给它原始 description 和用户的样本
  3. 让它用"给定这段 description，你会加载这个 skill 吗？理由？"提问
  4. 回读 agent 判断，反推 description 缺什么信号
- **输出形态**：
  ```
  负样本："cr 一下 PR !234"
  回放结论：agent 没触发本 skill，因为 description 没写 "!数字" / "PR 编号" 的信号
  建议：在 description 增加 `当用户提供 PR 编号（如 !234 或 #234）时触发`
  ```

---

## 执行顺序（分层执行）

**第一轮（默认跑 A/B/D/F）**：
静态扫四条 → 输出诊断结论 + 改写版 → 询问"是否需要跑 C / J 深度诊断？"

**第二轮（用户同意后）**：
- 同意 C：读目录做冲突扫描，追加冲突段 + 改写建议
- 同意 J：索要负样本，跑子 agent 回放，追加回放结论 + 改写建议

用户没同意就停住，不主动偷跑。

---

## 输出模板

```markdown
## 诊断结论

- [A 触发词覆盖度] ❌ 缺失：只写"代码审查工具"，没列用户口语说法
- [B 反向排除] ❌ 缺失：没有"不用于"子句
- [D 动作-对象结构] ⚠️ 部分：有"审查"动词，没有触发条件锚点
- [F 场景示例] ❌ 缺失：没有 utterance 样例

## 改写版

```yaml
description: |
  以资深工程师视角对代码变更做结构化审查。
  当用户说 "帮我 cr"、"review 一下"、"代码审查"，或提供 PR/MR 编号时触发 [A 新增][D 新增][F 新增]。
  example: 用户说 "cr 一下 !234" → 加载本 skill [F 新增]
  不适用于：单文件单行的快速质检（那种场景用 simplify）[B 新增]。
```

## 可选深度诊断

- C（跨 skill 冲突扫描）：是否扫 `~/.claude/skills/` 看有没有触发词冲突？
- J（负样本回放）：你有没有"应触发但没触发"或"不该触发但触发了"的原话？贴过来。
```

## 常见反模式

| 反模式 | 问题 | 修法 |
|---|---|---|
| `一个强大的 X 工具` | 纯营销语，零触发信号 | 改成"当用户 X 时触发" |
| `帮助用户 X 和 Y 和 Z...` | 能力过多，模型难抉择 | 拆 skill，或收窄 scope |
| `Use this skill when appropriate` | 把判断甩给模型 | 补充具体 utterance |
| 只写英文触发词 | 中文用户说中文时漏触发 | 中英文混列 |
| `IMPORTANT: ALWAYS trigger...` | 语气强但无条件 | 删掉 ALWAYS，写明条件 |
