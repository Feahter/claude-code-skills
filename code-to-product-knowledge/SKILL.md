# 从代码反向沉淀产品认知

把一个功能的**隐性产品认知**外化成显性文档。理论根是 SECI 模型的 Externalization——
通过逼问把脑子里"只可意会"的东西变成书面知识；操作上用 Feynman 自检：能用大白话讲清的就写，
讲不清的地方就是认知盲点，停下来逆问用户。

**这是 grill-me 的特化**：代码能确凿读出的（流程、规则、阈值、边界 case）自己读代码答；
代码读不出的盲区（痛点、目标用户、为何这样设计、视觉）停下来逐节逆问用户。

## 一条铁律：产品结论不从代码自作主张

代码里的实现细节 ≠ 产品意图。看到 `if (score > 0.85)` 不等于"产品规则是 0.85"——
它可能是临时 hack、可能有产品理由、可能早该改。**绝不把实现细节当成权威产品规则写进文档。**

文档里每条表述分两类标注：

- **事实**：代码里确凿能读出的，写明来源 `（src/xxx.ts:123）`。如"达到阈值 0.85 时触发提醒"。
- **推断 / 待确认**：需要解读意图才能下的结论，标 `🔶 推断` 或 `❓ 待确认`。
  如"🔶 推断：0.85 是为了过滤噪音信号——待确认"。

产品认知层面的结论（痛点、为何这样设计、目标用户）**只能来自用户确认**，不从代码臆测。

## 工作流

### 1. 锁定代码范围
用户给功能名或模块路径都接受：
- 给了**模块路径**→ 直接用。
- 给了**功能名**→ grep 业务词 / 查项目文档路由（如有 `CLAUDE.md` 路由表）定位候选入口，
  把候选文件清单报给用户确认后再开工。归属模糊（多候选、跨包）时停下问，不凭经验猜。

### 2. 分层探索（先骨架后精读）
- **第一轮**：委派 Explore agent 摸骨架——入口文件、主流程、状态机、关键文件清单、
  外部依赖（API/WS/SDK）。带上项目上下文，只要结论不要文件 dump。
- **第二轮**：主体针对骨架里的核心文件精读，补流程细节、条件判断、阈值常量、错误处理、边界分支。
- 别无脑全读：功能跨几十文件时尤其依赖第一轮 Explore 收敛范围。

### 3. 抽事实
按 know-what 模板（见 `references/know-what-template.md`）里**代码能答的节**先填初稿：
- ✅ 代码强项：核心功能描述（流程/规则/边界 case）、依赖与限制。
- 🟡 部分可推：概述的 What、埋点能看到的事件（但"成功标准"看不到）。
每条都按铁律标注事实 / 推断 / 待确认。

### 4. Feynman 自检 → 标盲点
逐段用大白话讲一遍。讲不通的点就是认知盲点，典型：
- 为什么这个阈值是这个值？想达成什么效果？
- 为什么这个场景要降级 / 特殊处理？
- 这个功能到底解决用户什么痛点？谁在什么情境下用？
把盲点按 know-what 的节归类，进入下一步。

### 5. 分批 grill（按节逆问）
用 `AskUserQuestion`，按 know-what 的节分批，**同一节的盲点一次问 2-4 个**，每题给推荐答案：
- 第一批：问题/背景（痛点）+ 目标用户与场景 —— 纯代码盲区。
- 第二批：核心逻辑里讲不清的规则 / 阈值 rationale。
- 第三批：为什么这样设计（权衡）+ 验证指标（成功标准）。
前一批答案会影响后一批怎么问——按需调整，不要一次性灌一份大问卷。

### 6. 生成文档
- 用 grill 答案把"推断/待确认"升级成确认结论，补齐代码盲区的节。
- **落盘前问用户路径**，并按项目线索给建议：项目有 `docs/<feature>/` 结构就建议
  `docs/<feature>/know-what.md`；有 AI 工作流产物约定（如 `.workflow/<task-id>/`）就提示该约定。
- 文档保留事实的来源标注；最后留"待确认/未解决"清单，便于后续 SECI 螺旋迭代。

## 输出文档结构

见 `references/know-what-template.md`（8 节：概述 / 问题背景 / 目标用户与场景 /
核心功能描述 / 视觉辅助 / 非功能要求 / 为何这样设计 / 验证与变更）。
重点不是写"是什么"，而是写"为什么"和"怎么用"。
