# 产品认知文档模板（know-what）

结合 SECI 外化与 Feynman 简单解释的结构化产品功能文档。重点写"为什么"和"怎么用"，
而非"是什么"。每条表述按主文件铁律标注：事实（带 `（文件:行）`）/ `🔶 推断` / `❓ 待确认`。

下表标了每节"代码能否反推"，指导 skill 哪些自己读、哪些必须 grill：

| 节 | 代码能反推吗 | 来源 |
|---|---|---|
| 1 概述 | 🟡 大概 | 代码 + 确认 |
| 2 问题/背景（痛点） | ❌ 盲区 | 必须 grill |
| 3 目标用户与场景 | ❌ 盲区 | 必须 grill |
| 4 核心功能描述 | ✅ 强项 | 代码为主 |
| 5 视觉辅助 | ❌ 代码外 | 截图/补充 |
| 6 非功能要求 | 🟡 部分 | 代码 + 确认 |
| 7 为何这样设计 | ❌ 除非有注释 | 必须 grill |
| 8 验证与变更 | 🟡 埋点可见、标准看不到 | 代码 + grill |

---

## 1. 标题与概述
功能名称、一句话描述（What + Why）。

## 2. 问题 / 背景
这个功能解决什么用户痛点 / 业务问题？认知来源（观察、反馈、经验）。

## 3. 目标用户与使用场景
谁用？在什么情境下用？（User Story / Journey）

## 4. 核心功能描述（Feynman 风格，用大白话）
- **它怎么工作**：流程、步骤、输入输出（可配流程图）。
- **关键逻辑 / 规则**：决策树、状态机、阈值（每个阈值标来源，rationale 待 grill 确认）。
- **边界情况 & 异常处理**：经验最有价值的地方。

## 5. 视觉辅助
截图、流程图、原型、演示。（代码外，需用户补充或标 ❓ 待补）

## 6. 非功能要求
性能、限制、依赖（外部 API / WS / SDK，代码可读；体感性能需确认）。

## 7. 为什么这样设计
决策 rationale 与权衡。代码里读不到（除非有注释），核心靠 grill。

## 8. 验证与变更历史
如何判断成功？已知问题、未来改进、变更记录。末尾附"待确认 / 未解决"清单。
